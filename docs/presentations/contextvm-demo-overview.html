Slide 2 / architecture

The plumbing is layered, not reinvented.

Browser or CLI client shell input, resize, interrupt, reconnect
v
Client-side bridge local Bun server for browser, direct MCP client for CLI
v
ContextVM over Nostr encrypted Nostr events, allowed pubkeys, skew-tolerant client subscription
v
Remote gateway ContextVM gateway launching the existing stdio MCP shell server
v
tmux-backed shell runtime durable shell session with `session_*` tools and poll/ack snapshots
one backend contract multiple client surfaces

Slide 3 / demo path

What we can demo right now

Remote CLI shell

Proven from a separate client machine using local `strfry` on the server plus `ssh -L`.

Browser terminal UI

Works locally and now has a ContextVM-backed browser bridge for the remote shell path.

Server scripts/contextvm-strfry-relay.sh start scripts/contextvm-private-demo.sh setup \ --server-relay-url ws://127.0.0.1:10549 \ --client-relay-url ws://127.0.0.1:10549 Client ssh -N -L 10549:127.0.0.1:10549 <user>@<server> CLI bun run demo:contextvm:interactive Browser bun run start:browser:contextvm
good proof commands: `hostname`, `pwd`, `uname -a` interrupt / close / reconnect all exist

Slide 5 / limits

What is still rough

Terminal fidelity

  • `tmux send-keys` is practical, not ideal
  • deep TUI edge cases still need harder testing
  • browser rendering is still poll/snapshot based

Deployment shape

  • browser still uses a local Bun bridge
  • file transfer is not first-class yet
  • same-key multi-client behavior needs clearer rules
the architecture works the next work is mostly fidelity and UX

Slide 6 / target

What we ideally want for the demo

Near-term ideal

  • CLI and browser both driving the real remote shell
  • clean reconnect story
  • simple bootstrap for operator and viewer
  • stable interrupt / close / reopen behavior

Longer-term ideal

  • better PTY fidelity than `tmux send-keys`
  • incremental or push-assisted terminal updates
  • browser-native auth/signing if worth it
  • structured file upload / download

The key message for the demo: we already proved the remote shell model over ContextVM. What remains is making it feel smoother, safer, and more product-shaped.

csh is past “can this work?” it is now in “how polished should the demo be?”
