csh / ssh over nostr / contextvm

Remote shell over ContextVM, with a browser path now in play.

We are proving an SSH-like shell flow over Nostr using ContextVM. Today the project has a working remote interactive shell demo, a working browser terminal UI, and a working browser-to-ContextVM bridge. The demo focus is not raw SSH packet parity. It is a durable shell session model with authenticated access, reconnectable state, and a clean path to browser use.

Working Demo Topology

`strfry` on the server, `ssh -L` relay forwarding to the client, ContextVM gateway on the server, interactive client and browser bridge on the client.

What We Can Show Right Now
  • remote interactive shell from a client machine over ContextVM
  • browser terminal UI backed by the same `session_*` contract
  • remote `Interrupt`, `Close`, and `Reconnect` flows

Architecture And Plumbing

The shell backend is still one stable contract. We are layering transports and clients around it rather than replacing it.

1. Browser or CLI client

Either the interactive terminal demo or the browser terminal UI initiates shell control through `session_open`, `session_write`, `session_resize`, `session_signal`, `session_poll`, and `session_close`.

2. Client-side bridge

CLI uses a direct ContextVM MCP client. Browser currently talks to a local Bun HTTP bridge, which can connect to a local stdio MCP server or to the remote server over ContextVM.

3. ContextVM over Nostr

The remote path uses a private gateway, encrypted Nostr events, whitelisted pubkeys, and a skew-tolerant client subscription lookback to avoid dropping valid relay responses.

4. Remote gateway and shell runtime

The gateway launches the existing stdio MCP shell server, which runs tmux-backed shell sessions and exposes deterministic poll/ack screen snapshots.

Current Feature Set

  • private pubkey-gated shell sessions over ContextVM
  • interactive terminal input, resize, interrupt, close, and reconnect

Current Limits

  • shell input still rides `tmux send-keys`, which is practical but not ideal PTY fidelity
  • browser rendering is still snapshot/poll based, not true incremental terminal streaming
  • the browser path still uses a local Bun bridge instead of browser-native ContextVM
  • same-key multi-client concurrency still needs clearer hardening/documentation
  • file upload/download is not yet a first-class browser capability

What We Ideally Want For Demo-Quality SSH Over Nostr

Near-Term Ideal

  • browser and CLI both talking to the real remote shell over ContextVM
  • stable interactive typing for common terminal workflows
  • clean reconnect and session recovery story
  • simple one-command demo bootstrap for server and client
  • demo-safe private topology with a reliable relay path

Longer-Term Ideal

  • better PTY fidelity than `tmux send-keys` for deeper TUI compatibility
  • incremental or push-assisted terminal updates where transport behavior allows it
  • browser-native auth/signing when the deployment model justifies it
  • structured file upload/download for browser users
  • latency and reconnect polish that gets closer to mosh-like resilience

The important framing for the demo: the project already proves the hard part, which is a real authenticated remote shell model over ContextVM. The next steps are about fidelity, UX, and deployment shape, not whether the architecture works at all.
